# 项目详情案例化设计

## 目标

把 macOS 装机助手和 API Pulse 的中英文详情页从简短说明升级为可快速扫描、内容可信的项目案例，同时在主图之后展示角色、状态、技术栈和结果摘要。

## 成功标准

- 四个项目详情页在主图之后、正文之前显示一致的项目事实区。
- 中文事实区使用“角色 / 状态 / 技术栈 / 结果”，英文使用 “Role / Status / Technologies / Outcome”。
- 英文页面把 `已完成产品` 显示为 “Completed product”，不直接暴露中文枚举值。
- 四份项目正文都有清晰的背景、问题、方案、关键实现或安全边界、结果等章节。
- 所有新增文字只能来自现有 frontmatter 和正文事实，不添加用户数、性能提升、节省时间等未经验证的指标。
- 卡片、主图、文章页面、URL、SEO 和已有图片行为不改变。
- 桌面与移动端可读，无横向溢出，自动化测试和发布验证通过。

## 方案选择

### 采用：复用组件加结构化 Markdown

新增 `ProjectFacts.astro`，接收项目内容条目和 locale，负责事实区的标签、本地化状态和语义化 `<dl>`。四份 Markdown 继续负责项目叙事章节。

该方案把重复的中英文事实展示集中在一个组件中，同时保留 Markdown 最适合编辑叙事内容的优势。

### 不采用：路由内重复事实区

在中英文动态路由分别编写 `<dl>` 虽然少一个文件，但会复制结构、标签和状态翻译，后续容易产生差异。

### 不采用：案例章节全部 schema 化

为背景、问题、方案等增加专用 schema 字段会使内容模型复杂化；当前只有两个项目，Markdown 标题已经能稳定表达结构。

## 组件设计

新增：

`src/components/ProjectFacts.astro`

接口：

```ts
interface Props {
  project: CollectionEntry<'projects'>;
  locale: 'zh' | 'en';
}
```

组件输出一个带 `.project-facts` 类的 `<dl>`，包含四项：

1. `role`
2. 本地化后的 `status`
3. `technologies`，复用 `.tags` 列表样式
4. `outcome`

只翻译现有三个状态枚举：

- `已完成产品` → `Completed product`
- `团队项目` → `Team project`
- `概念设计` → `Concept design`

中文保持原枚举值。组件不修改内容数据，也不引入新的 schema 字段。

## 页面接入

中英文项目动态路由在 `ContentLayout` 的默认 slot 中先渲染 `ProjectFacts`，再渲染 Markdown `Content`。由于 `ContentLayout` 已在 slot 前渲染摘要和主图，最终顺序为：

1. 日期
2. 标题与摘要
3. 项目主图
4. 项目事实区
5. 案例正文

文章路由不传入 `ProjectFacts`，保持现状。

## 内容结构

### macOS 装机助手

中文和英文正文采用相同信息结构：

- 背景：重复配置 Mac 的场景。
- 问题：软件选择、安全校验、安装和结果查看需要形成可检查的连续流程。
- 方案：SwiftUI 负责选择、预览和结果；Shell 负责校验与安装。
- 关键实现：dry-run、包校验、安装后验证、日志、APFS 写时复制和普通复制回退。
- 结果：形成可维护的本地装机工作流。

### API Pulse

中文和英文正文采用相同信息结构：

- 背景：需要同时查看 DeepSeek 余额、MiMo Key 状态和本地请求用量。
- 问题：汇总数据时不能把 API Key 暴露给浏览器或公共访客。
- 方案：服务端调用官方接口，本地 SQLite 汇总请求、Token、成功率和估算费用。
- 安全边界：Key 只在服务端内存使用；访客以 HttpOnly 会话 Cookie 隔离；公共部署默认不读取站点所有者的本地凭据。
- 结果：在既定隐私边界内统一呈现余额和本地用量。

不新增现有内容未支持的技术、指标、客户、规模或效果描述。

## 样式与响应式

- `.project-facts` 使用带边框的网格容器，与现有深色表面和圆角变量一致。
- 桌面端四项按两列排列；移动端改为单列。
- 每个事实项使用 `<dt>` 作为短标签、`<dd>` 作为内容。
- 技术栈继续使用现有 `.tags`，不创建第二套标签样式。
- 不改变 `.prose` 最大宽度、主图比例或全站卡片样式。

## 测试

按测试驱动方式先增加失败 E2E 测试：

1. 中文项目详情显示四个中文标签和对应内容。
2. 英文项目详情显示四个英文标签，状态为 “Completed product”。
3. 两个项目中英文正文显示约定的章节标题。
4. API Pulse 的安全边界文字和 macOS 的关键实现文字保持可见。
5. 移动端事实区为单列且页面无横向溢出。
6. 文章详情页不出现 `.project-facts`。

实现后运行完整单元测试、Astro Check、生产构建、build audit、全部 E2E、两次依赖审计和生产隐私扫描。

实现合入 `main` 后直接推送。等待 GitHub Pages 工作流成功，再用线上 DOM 验证中英文标签、英文状态翻译、案例标题、移动端布局以及文章页面隔离。

## 非目标

- 不新增项目、图片或文章。
- 不修改 frontmatter 数据和 schema。
- 不虚构量化成果或外部背书。
- 不重做项目卡片、主图、导航或整体视觉系统。
