# 项目证据闭环设计

## 目标

在不公开源码、安装包、可执行文件或内部配置的前提下，为 macOS 装机助手与 API Pulse 的中英文项目详情页增加可追溯的所有者验证记录。访客应能看出项目验证了什么、何时验证、在哪类环境中验证，以及哪些材料刻意保持私有。

本阶段的成功标准是：四个项目详情页均展示本地化证据区；同一项目的中英文版本、验证日期和检查数量一致；所有公开结论均来自本次实际执行的检查；构建、隐私审计和线上验收全部通过。

## 范围

### 包含

- 在项目内容模型中增加结构化证据数据。
- 为两个项目执行一次不泄露凭据的真实验证。
- 在中英文项目详情页内嵌共享证据组件。
- 使用现有产品截图，不新增日志或本地环境截图。
- 增加 schema、内容一致性、页面呈现和隐私回归测试。
- 合并、推送并复验 GitHub Pages 线上结果。

### 不包含

- 公开或新建项目源码仓库。
- 提供安装包、可执行文件或脱敏体验包下载。
- 部署 API Pulse 后端或公开在线演示。
- 制作演示视频或 PDF 报告。
- 修改两个原项目的产品功能、安全逻辑或发布流程。
- 把所有者验证记录描述为第三方安全认证。

## 证据来源与真实性边界

网站内容修改前，先从两个本地项目读取真实版本并执行当前可用的验证命令。

macOS 装机助手的候选证据包括本地完整性检查、全量本地 dry-run、校验和验证以及项目现有测试。API Pulse 的候选证据包括 Node.js 语法检查、临时本地启动、HTTP 行为检查、访客会话隔离和浏览器输出不包含密钥。实施计划必须在读取两个项目的当前脚本和接口后确定准确命令，不能凭名称推断行为。

只有在本次运行中退出成功、且输出与声明一致的检查，才可以写入 `checks`。验证日期使用实际执行日期。任何失败都必须如实记录并定位；本阶段不顺手修改原项目，也不能把失败或未执行项目写成“已通过”。若失败使批准的公开证据不足，停止网站实现并报告阻塞。

公开证据不得包含：

- API Key、Token、密码、Cookie 或其他凭据。
- `/Users/...` 等本地绝对路径。
- 内部配置值、访问锁内容或本机数据库内容。
- 第三方安装包文件名、包清单或可下载地址。
- 未经本次验证支持的性能、安全或兼容性结论。

## 内容模型

项目 schema 增加必填的 `evidence` 对象：

- `version`：本次验证对应的版本或明确的项目版本标识。
- `verifiedAt`：`YYYY-MM-DD` 格式的实际验证日期。
- `environment`：适合公开的验证环境概述，不包含设备名、用户名或绝对路径。
- `checks`：至少一项本次实际通过的本地化检查说明。
- `disclosure`：本地化的公开边界说明，明确源码和内部材料保持私有。

每个中英文 Markdown 文件保存自己的本地化 `checks`、`environment` 和 `disclosure`。同一 `translationKey` 下的中英文内容必须保持 `version`、`verifiedAt` 和 `checks` 数量一致。这样避免在组件中硬编码项目文案，同时用测试防止可验证事实漂移。

## 页面与组件

新建共享 `ProjectEvidence.astro` 组件，接收项目条目与语言，不读取外部目录或运行验证命令。组件只负责把已经验证并写入内容集合的数据渲染为语义化 HTML。

证据区位于现有 `ProjectFacts` 之后、Markdown 案例正文之前，标题为“验证记录”或 “Verification Record”。内容顺序为：

1. 版本、验证日期、验证环境组成的事实网格。
2. 本次已通过检查的列表。
3. 私有材料与认证边界说明。

日期使用 `<time>`，事实使用描述列表，检查使用普通列表。桌面端采用紧凑网格，移动端降为单列。样式复用现有颜色、边框、圆角和间距变量；不增加客户端 JavaScript、动态折叠、徽章系统或新的设计抽象。

现有两张产品截图继续作为项目主图。证据区不展示测试日志截图，避免泄露路径、配置或本地状态。

## 失败处理

- schema 缺少任一证据字段、日期格式非法或检查列表为空时，构建失败。
- 同一项目的中英文版本、日期或检查数量不一致时，单元测试失败。
- 本地项目验证失败时，不先写网站内容；保留原项目状态并报告准确失败项。
- 页面隐私扫描发现敏感模式时，停止提交和部署。
- 线上证据区与本地构建不一致时，不宣告完成，继续检查部署提交和 CDN 内容。

## 测试与验收

### 数据测试

- schema 接受完整证据对象。
- schema 拒绝缺失字段、非法日期和空检查列表。
- 中英文配对项目的 `version`、`verifiedAt` 和检查数量一致。

### 页面测试

- 四个中英文项目详情页展示对应标题、版本、日期、环境、检查和披露说明。
- 证据区位于项目概况之后、案例正文之前。
- 390px 视口下无横向溢出，内容保持可读。
- 页面不出现下载按钮、源码仓库链接或第三方认证措辞。

### 隐私与发布验证

- 运行两个原项目的批准验证命令，并保存仅用于本次实施判断的输出摘要。
- 运行个人网站完整 `npm run verify`。
- 执行针对绝对路径、凭据模式、内部配置和第三方包名的隐私扫描。
- 合并到 `main` 并推送，等待对应提交的 GitHub Pages 工作流成功。
- 在线检查四个项目页的证据内容、移动端结构和敏感信息缺失情况。

## 维护规则

验证记录是时间点事实。原项目版本或检查结论发生变化时，必须重新运行验证，再同步更新版本、日期和检查列表。仅修改文案或日期而不重新执行检查是不允许的。
