# 关于页能力与可信度设计

## 目标

把中英文关于页从简短履历摘要升级为能够支持合作判断的能力说明页。访客应能快速理解 WKL 在做什么、如何处理问题、哪些真实项目可以作为证据，以及如何直接联系。

## 成功标准

- 中英文关于页明确聚焦 macOS 工具与自动化、API 产品实践。
- 页面展示三个能力方向和三个工作步骤，内容具体但不夸大。
- 两个代表项目从现有 `projects` 内容集合读取，不复制项目名称、结果或链接数据。
- 页面提供公开邮箱主入口和当前语言项目列表的次入口。
- 中英文页面结构和信息层级一致。
- 不展示求职状态，不添加未经验证的量化成果，不重复首页的三项课程或团队经历。
- 390px 与桌面视口无横向溢出，标题层级、语义链接和键盘焦点保持可用。
- 既有测试、Astro 检查、构建审计和部署流程无回归。

## 信息结构

### 首屏介绍

保留现有 `.prose` 页面宽度、眉题和单一 `h1`。引导文案不再只描述“通过项目学习”，而是明确说明当前实践集中在 macOS 工具与自动化、API 产品，并强调可靠实现、安全边界和可复用记录。

### 能力方向

使用一个语义列表展示三个能力组：

1. **工具与自动化**：SwiftUI、Shell、macOS 工作流与安装流程。
2. **API 与本地数据**：Node.js、HTTP API、SQLite 与本地状态管理。
3. **安全边界与可维护性**：输入验证、隐私边界、可回滚流程、测试和文档。

这些内容是当前实践范围，不使用“专家”“精通”等无法验证的等级描述。

### 工作方式

使用有序列表表达三个稳定步骤：

1. 从真实问题和使用路径出发。
2. 先明确安全、隐私和失败边界。
3. 将方案实现为可维护、可测试并有记录的工具或产品。

英文页提供含义一致的英文文案，不引用中文经历内容。

### 代表项目证据

关于页在构建时读取当前语言下 `featured: true` 的两个项目。每项只展示：

- 项目标题；
- `outcome` 结果摘要；
- 指向当前语言项目详情页的链接。

项目排序沿用内容集合返回顺序，不在关于页创建第二套排序字段。中文详情链接使用项目 `id`，英文链接使用 `translationKey`，与现有首页保持一致。

关于页不渲染完整 `ProjectCard`，避免视觉重复和大图占据个人说明页；使用轻量证据列表即可。

### 联系行动区

页面末尾使用与首页相同的 `.hero-actions`、`.action-link` 和 `.action-link-primary` 样式：

- 主入口读取 `site.email` 并显示完整邮箱；
- 次入口进入当前语言项目列表；
- 不出现“开放机会”“求职中”等状态文本。

## 实现边界

- 修改 `src/pages/about.astro` 和 `src/pages/en/about.astro`，在各自页面读取对应语言的精选项目。
- 在 `src/styles/global.css` 增加轻量的能力列表、工作方式和项目证据列表样式。
- 不新增内容集合、schema 字段、运行时脚本、表单、第三方服务或依赖。
- 不修改首页、项目详情、导航、页脚或部署配置。
- 两页结构简单且仅有本地化文案差异，直接在页面模板中实现，不创建只被两页使用且不能减少复杂度的新组件。

## 数据流与失败行为

Astro 在构建时通过 `getCollection('projects')` 读取项目，按 `lang` 和 `featured` 过滤。页面只消费现有 schema 已要求的 `title`、`outcome`、`translationKey` 和项目 `id`。

内容集合 schema 已保证这些字段存在，因此不增加运行时兜底。若未来没有精选项目，语义列表自然为空，构建仍可完成；本轮 E2E 明确要求当前两个项目都出现，防止内容配置意外退化。

## 响应式与可访问性

- 能力组在宽屏可使用多列，在 48rem 以下回到单列。
- 工作步骤和项目证据保持正常文档流，不依赖横向滚动。
- 长邮箱使用现有 `overflow-wrap` 行为并在移动端占满可用宽度。
- 每页只有一个 `h1`；“能力方向”“工作方式”“代表项目”“联系”使用 `h2`；项目标题使用 `h3`。
- 交互元素均为原生链接，继续使用全局 `:focus-visible` 样式。

## 验证方案

- 新增关于页 E2E 测试，验证中英文 `h1`、三个能力项、三个有序步骤和两个代表项目。
- 验证项目详情链接、`mailto:` 链接和项目列表链接均指向正确语言路径。
- 验证页面不包含求职状态文案，也不渲染 `.experience-list`。
- 在 390px 检查无横向溢出、行动链接可见且可聚焦。
- 运行 `npm run verify`、两种依赖审计和生产目录隐私扫描。
- 本地目检中文桌面端与英文移动端。
- 推送后等待 GitHub Pages 成功，并在线复验关键 DOM、链接和响应式状态。
