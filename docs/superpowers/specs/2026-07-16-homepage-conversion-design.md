# 首页可信度与联系转化设计

## 目标

让首次访问者在首页快速理解 WKL 当前实践的方向、看到可信的项目证据，并能直接通过公开邮箱联系。首页定位聚焦 macOS 工具与自动化、API 产品实践，不展示求职或机会状态。

## 成功标准

- 中英文首页首屏都明确传达 macOS 工具、自动化与 API 产品实践的定位。
- 首屏提供公开邮箱的主行动入口和项目列表的次行动入口。
- 两种语言使用一致的信息层级和行动路径。
- 精选项目仍以现有内容集合为唯一数据来源，不在首页复制项目事实。
- 英文首页补齐可用的内容入口，但不伪造文章或经历的英文译文。
- 桌面端与移动端无横向溢出，键盘焦点、标题层级和语义链接保持可用。
- 既有单元测试、Astro 检查、构建审计和端到端测试无回归。

## 页面结构

### 首屏

保留现有 `.hero` 和超大标题视觉语言，只调整内容并增加行动区域：

1. 眉题继续表达“个人实验室 / Personal Lab”。
2. 主标题强调把真实问题做成可维护的工具与产品。
3. 引导文案明确列出 macOS 工具与自动化、API 产品实践两个方向。
4. 主行动链接使用 `mailto:2694421597@qq.com`，链接文本同时公开邮箱。
5. 次行动链接进入当前语言的项目列表：中文 `/projects/`，英文 `/en/projects/`。

页面不出现“正在寻找机会”“开放求职”或类似状态文案。

### 精选项目

保留现有项目卡及内容查询方式，在章节标题附近加入一句简短说明，告诉访客这些项目展示了从问题识别、安全边界到可维护实现的完整过程。

项目名称、摘要、状态、技术栈和结果继续来自 `projects` 内容集合。首页不创建第二份项目数据，也不把详情页的事实卡复制到首页。

### 后续内容

中文首页继续展示精选项目、最新文章、其他经历和关于入口。

英文首页保持相同的主要信息顺序。精选项目之后增加两个轻量入口：以 “Notes in Chinese” 明确标注并链接到现有中文文章列表 `/posts/`，以 “Experience” 链接到已有英文履历摘要 `/en/about/`。英文首页不把中文文章或经历包装成英文卡片，也不伪造译文；完整翻译留给后续独立工作。

## 实现边界

- 修改中英文首页文案和结构。
- 在现有全局样式中增加轻量的行动链接组样式，并复用已有颜色、边框和圆角变量。
- 如中英文首页出现完全相同的行动链接结构，可提取一个只负责渲染链接的简单组件；若直接模板重复更少，则不创建组件。
- 不修改项目详情页、内容集合 schema、导航结构或部署配置。
- 不新增表单、邮件服务、访客统计、动画库或新的运行时依赖。

## 响应式与可访问性

- 行动链接在宽屏横向排列，在窄屏自然换行或纵向排列。
- 邮箱地址允许换行，不应造成 390px 视口横向滚动。
- 链接保留清晰的键盘焦点样式；主次行动不能只依赖颜色区分。
- 页面继续使用单一 `h1`，各内容区使用 `h2`，项目卡标题保持 `h3`。
- 中英文文本、`lang` 属性和语言切换路径保持一致。

## 数据流

首页继续通过 Astro `getCollection()` 在构建时读取项目、文章和经历。新增首屏文案和行动链接是静态本地化内容，不引入客户端状态或请求。邮箱继续读取 `src/lib/site.ts` 的 `site.email`，避免在多个页面新增不同的邮箱来源。

## 验证方案

- 为中英文首页新增端到端断言：定位文案、公开邮箱链接、项目入口和标题层级。
- 验证英文首页只提供明确标注的中文文章入口和英文履历入口，不渲染不存在的英文文章或经历卡片。
- 在 390px 与桌面视口检查无横向溢出，行动链接可见且可聚焦。
- 运行 `npm run verify`、两种依赖审计和生产目录隐私扫描。
- 本地截图目检中英文首页的桌面端和移动端。
- 推送后等待 GitHub Pages 部署成功，并在线检查首页关键 DOM 与链接。
