# WKL 个人网站设计规格

## 目标

建立一个兼顾求职展示与长期写作的个人网站。网站以真实项目和解决问题的过程为核心，不限定单一岗位方向；中文为主要内容语言，首页、关于与履历摘要提供中英文版本。

首版成功标准是：访客能在一分钟内理解 WKL 在做什么、看完两个代表项目，并找到文章、经历与联系方式；维护者可以通过新增 Markdown/MDX 文件发布项目或文章。

## 设计原则

- 使用最少的运行时 JavaScript，优先静态生成。
- 区分已完成产品、团队项目和概念设计，不夸大个人贡献。
- 不公开学号、教师姓名、团队成员名单、密钥、本地路径或内部文件内容。
- 首版公开联系方式为 `2694421597@qq.com`。
- 首版不做后台、登录、评论、搜索、自动翻译和复杂动画。
- 视觉借鉴 Gothsec 的清晰结构与 Magic Portfolio 的内容层级，但组件和样式独立实现。

## 技术路线

使用 Astro、TypeScript 和 Astro Content Collections 从零搭建静态站点。选择该路线是因为它适合内容型网站、客户端负担低、部署平台无关，也避免继承 Magic Portfolio 的非商业许可限制。

站点由以下边界清晰的模块组成：

- `BaseLayout`：页面骨架、元数据、跳转链接、导航和页脚。
- 展示组件：项目卡片、文章卡片、经历条目、标签和语言切换。
- 内容集合：`projects`、`posts`、`experience`，由 schema 在构建阶段校验。
- 路由页面：中文默认路由与 `/en/` 英文核心路由。
- 设计令牌：颜色、字体、间距、圆角、阴影和动效时长集中管理。

只有语言切换等必要交互使用少量客户端脚本。图片通过 Astro 图片能力生成响应式尺寸；字体优先使用本地或系统字体。

## 信息架构

首版包含五类页面：

1. 首页：个人定位、两项精选项目、最新文章、简短经历和联系方式。
2. 项目列表与详情：装机助手、API Pulse 为独立详情；其他成果仅简述。
3. 文章列表与详情：中文文章为主，提供标签、日期和阅读时长。
4. 关于页：昵称、头像或字标、学习方向、能力关键词、页面式履历摘要和公开邮箱。
5. 英文核心页：首页、关于页、履历摘要，以及重要项目的英文摘要。

首版不提供可下载的简历文件；履历信息直接展示在关于页，避免使用尚未审阅的本地简历材料。

## 首版内容

### 代表项目

- macOS 装机助手：原生 SwiftUI 与 Shell 组成的本地批量装机工具，重点讲实际需求、安全检查、可逆操作和持续迭代。
- API Pulse：DeepSeek 与 Xiaomi MiMo API 用量仪表盘，重点讲接口整合、服务端密钥边界、会话隔离和部署限制。

### 其他经历

- 全国大学生统计建模大赛：城市社会工作服务评价，简述数据采集、AHP/EWM 模型和城市实证分析，并标记为团队项目。
- 智庆余年：智慧养老与 3D 打印适老化方案，简述用户研究与产品策划，并标记为团队项目。
- OmniGuide：多模态视障辅助系统，简述端边云构想与技术伦理，并明确标记为课程概念设计。

### 首发文章

- 《我如何把 macOS 批量安装流程做成原生工具》
- 《API Key 仪表盘中的隐私边界》

文章根据现有项目事实重新整理，不复制含个人信息的原始文档。

## 内容模型

`projects` 字段包括：标题、摘要、角色、日期、技术、封面、状态、成果、语言和正文。状态必须从“已完成产品”“团队项目”“概念设计”中选择。

`posts` 字段包括：标题、摘要、发布日期、标签、封面、语言、草稿状态和正文。

`experience` 字段包括：名称、日期、个人参与说明、单句成果、类型和语言。

必填字段缺失或值不合法时，构建必须失败，避免发布信息残缺的页面。

## 双语规则

中文是默认语言，英文核心内容位于 `/en/`。语言切换优先进入当前页面的对应译文；如果当前文章没有英文版，则进入英文首页并显示“此内容仅提供中文版本”的提示。站点不自动生成翻译，也不生成空白英文页面。

中文内容和英文内容通过稳定的翻译键关联。普通中文文章不要求英文版，重要项目可以只提供英文摘要。

## 视觉与交互

视觉方向为“温暖科技感”：暖黑背景、米白正文、橙色单一强调色。首页用编辑式大标题突出“把好奇心，做成看得见的东西”。初始品牌名称使用 `WKL`；若没有已确认头像，则使用 `WKL` 字标，不使用随机人物图片。

项目卡片采用大图、短摘要和少量技术标签。桌面端强调留白与内容层级，移动端使用单列布局。动效仅包括卡片悬停、页面淡入和导航状态变化，并遵循 `prefers-reduced-motion`。

交互必须支持键盘操作、明显的焦点样式、足够的文字对比度和图片替代文本。

## 异常与回退

- 缺少必填内容：构建失败并指出对应文件与字段。
- 缺少英文译文：按双语规则回到英文首页并提示，不自动翻译。
- 缺少封面：使用统一品牌占位图。
- 路径不存在：展示中英文 404 页面，并提供返回首页入口。
- 外部链接：在内容录入和构建检查中验证格式；无效链接不得进入首版。

## 部署边界

首版保证本地开发、生产构建与标准静态部署兼容，不绑定具体托管商。之后可以部署到 GitHub Pages、Cloudflare Pages 或其他静态平台；部署账号、域名和统计服务不属于本次首版范围。正式站点地址确定前不输出 canonical，避免把本地构建地址发布为规范链接。

## 验证与验收

实现完成时必须满足：

- 首页、项目、文章、关于、英文核心页面和 404 页面均可访问。
- 两个代表项目、三项简要经历和两篇文章正确展示。
- 内容 schema 能拒绝缺少必填字段或状态错误的内容。
- 中文与英文核心路由可互相切换；无英文文章按规则回退。
- 生产构建、类型检查和内部链接检查通过。
- 桌面与手机常见宽度无明显错位，主要路径可通过键盘完成。
- 页面源文件和生成结果不包含学号、密钥、本地绝对路径或未批准的个人信息。
- 基础无障碍检查通过；Lighthouse 的性能、无障碍、最佳实践和 SEO 以 90 分以上为优化目标，但不为追分增加复杂架构。

## 不在首版范围内

后台管理、用户系统、评论、全文搜索、访问统计、邮件订阅、自动翻译、动态数据库、在线简历生成器、自定义域名和 GitHub 自动部署均留待后续按实际需求决定。
